Monitoring Guide

This daemon checks all configured endpoints every 15 minutes. Each check produces a value (what was measured) and a state (OK, WARNING, CRITICAL, or NO_DATA). State changes trigger notifications via Discord, Telegram, email, or text file.

Core Concepts

Each monitored endpoint is defined by these fields:

Name Display name for the service (e.g. KernelCI Production API).
Sub-name Optional label to distinguish multiple checks on the same service. Use sub-names like HTTP, Latency, Cert, Disk Usage when you monitor the same service in several ways.
Endpoint The URL or connection target to check (always a real URL, or a namespace name for Kubernetes).
Check Type Determines which checker runs: http_status, http_latency, tls_cert, prometheus, postgresql, kubernetes, or docker.
Selector What to extract from the response. Meaning depends on the check type. Leave empty when not applicable.
Condition Alert rule applied after measurement. If the value matches the condition, state worsens to WARNING. Leave empty for default behavior only.
Critical Flag for production-critical services. These get [CRITICAL] prefix in notifications.
Enabled Toggle monitoring on/off without deleting configuration.

States

OKService is healthy. HTTP 2xx, connection succeeded, value within threshold.
WARNINGCondition triggered, or a non-fatal issue (e.g. short uptime, cert expiring soon).
CRITICALService unreachable, HTTP 5xx, connection failed, check timed out (30s).
NO DATANo checks have run yet, or the metric was not found.

Makes a GET request and records the HTTP status code. State mapping: 2xx → OK, 5xx → CRITICAL, other → WARNING.

Prometheus Metrics

Check type: prometheus

Scrapes a /metrics endpoint directly on each service (not a Prometheus server). Set the Endpoint to the full metrics URL. Parses the Prometheus text exposition format and extracts a specific metric value.

Selector syntax

(metric_name)
(metric_name,label=value)
(metric_name,label1=value1,label2=value2)

If Selector is empty, only checks that the metrics endpoint is reachable.

Examples

What to monitorEndpointSelectorCondition
Disk usage % https://storage.kernelci.org/metrics (node_filesystem_avail_bytes,mountpoint=/) <10000000000
Inode usage % https://storage.kernelci.org/metrics (node_filesystem_files_free,mountpoint=/) <100000
HTTP request rate https://api.kernelci.org/metrics (http_requests_total,status=500) diff(12h)>1000
Pipeline jobs pending https://pipeline.kernelci.org/metrics (pipeline_jobs_pending) >100
Pipeline stalled https://pipeline.staging.kernelci.org/metrics (pipeline_last_job_completed_timestamp) diff(12h)==0
Nginx 5xx rate https://dashboard.kernelci.org:9113/metrics (nginx_http_requests_total,status=5xx) diff(12h)>500
Memory usage https://storage.kernelci.org/metrics (node_memory_MemAvailable_bytes) <1000000000

Tips

  • Use diff(12h)>N to detect counter spikes — useful for error rates and request counters that only go up.
  • Use diff(12h)==0 to detect stalled counters (e.g. pipeline stopped processing jobs).
  • Use absdiff(12h)>N to detect any significant change in either direction — useful for values that should stay stable.
  • Label filters are exact match. For multiple labels, separate with commas.

PostgreSQL

Check type: postgresql. Endpoint: postgresql://user:password@host:port/database

Connects to the database and optionally runs a query. Connection failure → CRITICAL.

Examples

What to monitorEndpointSelectorCondition
Basic up/down postgresql://user:pass@db.kernelci.org/kcidb — —
Read-only check postgresql://user:pass@db.kernelci.org/kcidb SELECT pg_is_in_recovery()::int ==1
Table row count postgresql://user:pass@db.kernelci.org/kcidb SELECT count(*) FROM submissions WHERE created_at > now() - interval '1 hour' ==0
Replication lag postgresql://user:pass@db.kernelci.org/kcidb SELECT EXTRACT(EPOCH FROM replay_lag)::int FROM pg_stat_replication LIMIT 1 >300
Active connections postgresql://user:pass@db.kernelci.org/kcidb SELECT count(*) FROM pg_stat_activity WHERE state = 'active' >50

The Selector is the SQL query. The first column of the first row is the measured value. Leave empty for a simple SELECT 1 connectivity check.

Kubernetes

Check type: kubernetes. Endpoint is the namespace name. Selector is the optional label filter.

Connects via the system kubeconfig. Lists pods and inspects each container's restart count and current state. The measured value is the sum of container restart counts across all matching pods.

  • CRITICAL if any container is in CrashLoopBackOff, ImagePullBackOff, ErrImagePull, CreateContainerError, CreateContainerConfigError, or InvalidImageName.
  • WARNING if any container has restarted in the last 30 minutes (based on lastState.terminated.finishedAt).
  • Otherwise the user-supplied condition is evaluated against the total restart count, so growth-based alerts like diff(12h)>5 work directly.

Examples

What to monitorEndpointSelectorConditionNotes
All production pods production — — All pods in production namespace
API pods only production app=api — Only pods with label app=api
Pipeline workers production component=pipeline-worker — Filter by component label
Staging pods staging — — Staging namespace, non-critical

What it measures

  • Value: total number of matching pods.
  • State: WARNING if any pod uptime < 30 minutes, OK otherwise.
  • The message lists which pods have low uptime, e.g. api-7f8b9-xk2j4: uptime 3m.
  • Uses the system kubeconfig (typically ~/.kube/config or in-cluster credentials).

Where to put the kubeconfig

The daemon loads credentials via the standard kube-rs lookup, in this order:

  1. $KUBECONFIG environment variable, if set.
  2. $HOME/.kube/config.
  3. In-cluster service-account token at /var/run/secrets/kubernetes.io/serviceaccount/ (used automatically when running inside a Kubernetes pod).

When installed from the .deb package, the service runs as user kernelci-status with home /var/lib/kernelci-status and systemd hardening (ProtectHome=yes, ProtectSystem=strict). The simplest option is:

sudo install -d -o kernelci-status -g kernelci-status -m 700 \
    /var/lib/kernelci-status/.kube
sudo install -o kernelci-status -g kernelci-status -m 600 \
    /path/to/your/kubeconfig /var/lib/kernelci-status/.kube/config
sudo systemctl restart kernelci-status

ProtectHome=yes only masks /home, /root, and /run/user — it does not affect /var/lib, so $HOME/.kube/config resolves correctly here.

Alternative — put it under /etc: drop the file at /etc/kernelci-status/kubeconfig (mode 0640, group kernelci-status) and edit the systemd unit:

[Service]
Environment=KUBECONFIG=/etc/kernelci-status/kubeconfig
ReadOnlyPaths=/etc/kernelci-status.toml /etc/kernelci-status/kubeconfig

Then systemctl daemon-reload && systemctl restart kernelci-status.

Gotcha: if the kubeconfig uses an exec: auth plugin (gke-gcloud-auth-plugin, aws-iam-authenticator, etc.), the plugin binary must be reachable on PATH for the kernelci-status user. ProtectSystem=strict keeps /usr/bin and /usr/local/bin readable, but binaries living under /home will be invisible to the service.

Docker via SSH

Check type: docker. Endpoint: ssh://user@host

SSHs into a remote host, runs docker ps, and parses container uptime. WARNING if any container has been running for less than 30 minutes. Uses the system SSH config (~/.ssh/config, keys, known_hosts).

Examples

What to monitorEndpointSelectorConditionNotes
All containers on host ssh://deploy@storage.kernelci.org — — Checks all running containers
Nginx only ssh://deploy@storage.kernelci.org nginx — Substring match on container name
Legacy storage ssh://deploy@legacy-storage.kernelci.org storage — Only containers with "storage" in name
KCIDB services ssh://deploy@kcidb.kernelci.org kcidb — All KCIDB-related containers

What it measures

  • Value: number of matching running containers.
  • State: WARNING if any container uptime < 30 minutes, CRITICAL if SSH connection fails or docker ps errors.
  • The message lists containers with low uptime.
  • SSH must be pre-configured — the daemon uses the system SSH agent and ~/.ssh/config.

Docker restart counts via Prometheus

To monitor container restart counts rather than just current uptime, expose metrics with cAdvisor or a Docker exporter and use a Prometheus check:

WhatEndpointSelectorCondition
Container restarts https://storage.kernelci.org:9101/metrics (container_restart_count,name=nginx) diff(12h)>3

Conditions & Alerts

Conditions are applied after the checker runs. They can only make the state worse (never improve it). If a checker already returns CRITICAL, a condition cannot downgrade it to WARNING.

Operators

ConditionTriggers WARNING whenExample use case
>NValue is greater than N>90 — disk usage above 90%
<NValue is less than N<7 — cert expires in less than 7 days
>=NValue is greater than or equal to N>=100 — connection pool full
<=NValue is less than or equal to N<=0 — no free space
==NValue equals N==1 — database in read-only mode
!=NValue does not equal N!=0 — error count is non-zero

Diff over time

Compare the current value against a historical measurement:

diff(12h)>100    — warn if value grew by more than 100 in the last 12 hours
diff(24h)<-50    — warn if value dropped by more than 50 in 24 hours
diff(12h)==0     — warn if value hasn't changed (stalled counter)

The daemon looks up the value recorded approximately that many hours ago (±30 minute tolerance window) and computes current - past.

Absolute diff over time

Like diff(), but uses the absolute value of the change — triggers regardless of direction (increase or decrease):

absdiff(12h)>100 — warn if value changed by more than 100 in either direction
absdiff(24h)>50  — warn if value drifted by more than 50 in 24 hours

Computes |current - past|. Useful when any significant change is suspicious, not just increases or decreases.
