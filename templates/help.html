Monitoring Guide

This daemon checks all configured endpoints every 15 minutes. Each check produces a value (what was measured) and a state (OK, WARNING, CRITICAL, or NO_DATA). State changes trigger notifications via Discord, email, or text file.

Core Concepts

Each monitored endpoint is defined by these fields:

Name Display name for the service (e.g. KernelCI Production API).
Sub-name Optional label to distinguish multiple checks on the same service. Use sub-names like HTTP, Latency, Cert, Disk Usage when you monitor the same service in several ways.
Endpoint The URL to check. The URL scheme determines the type of check — see sections below.
Selector What to measure from the endpoint. Meaning depends on the check type. Leave empty for the default measurement.
Condition Alert rule applied after measurement. If the value matches the condition, state worsens to WARNING. Leave empty for default behavior only.
Critical Flag for production-critical services. These get [CRITICAL] prefix in notifications.
Enabled Toggle monitoring on/off without deleting configuration.

States

OKService is healthy. HTTP 2xx, connection succeeded, value within threshold.
WARNINGCondition triggered, or a non-fatal issue (e.g. short uptime, cert expiring soon).
CRITICALService unreachable, HTTP 5xx, connection failed, check timed out (30s).
NO DATANo checks have run yet, or the metric was not found.

HTTP / HTTPS Monitoring

Endpoint scheme: https:// or http://

Three kinds of checks depending on the Selector:

1. Status code check (default)

Leave Selector empty. The daemon makes a GET request and records the HTTP status code.

NameEndpointSelectorCondition
KernelCI APIhttps://api.kernelci.org——
Dashboardhttps://dashboard.kernelci.org——

State mapping: 2xx → OK, 5xx → CRITICAL, other → WARNING.

2. Latency check

Set Selector to latency. Records response time in milliseconds.

NameSub-nameEndpointSelectorCondition
KernelCI APILatencyhttps://api.kernelci.orglatency>500
DashboardLatencyhttps://dashboard.kernelci.orglatency>2000

Combine with a condition like >500 to warn when response time exceeds 500ms.

3. TLS certificate expiration

Set Selector to cert_expiration. Connects via TLS, parses the leaf certificate, and records days until expiration.

NameSub-nameEndpointSelectorCondition
DashboardCerthttps://dashboard.kernelci.orgcert_expiration<7
GrafanaCerthttps://grafana.kernelci.orgcert_expiration<14

Built-in: WARNING if < 7 days, CRITICAL if expired. Add <14 for earlier warning.

Prometheus Metrics

Endpoint scheme: promhttps:// or promhttp://

Scrapes the /metrics endpoint directly on each service (not a Prometheus server). Parses the Prometheus text exposition format and extracts a specific metric value.

Selector syntax

(metric_name)
(metric_name,label=value)
(metric_name,label1=value1,label2=value2)

If Selector is empty, only checks that the metrics endpoint is reachable.

Examples

What to monitorEndpointSelectorCondition
Disk usage % promhttps://storage.kernelci.org (node_filesystem_avail_bytes,mountpoint=/) <10000000000
Inode usage % promhttps://storage.kernelci.org (node_filesystem_files_free,mountpoint=/) <100000
HTTP request rate promhttps://api.kernelci.org (http_requests_total,status=500) diff(12h)>1000
Pipeline jobs pending promhttps://pipeline.kernelci.org (pipeline_jobs_pending) >100
Pipeline stalled promhttps://pipeline.staging.kernelci.org (pipeline_last_job_completed_timestamp) diff(12h)==0
Nginx 5xx rate promhttps://dashboard.kernelci.org:9113 (nginx_http_requests_total,status=5xx) diff(12h)>500
Memory usage promhttps://storage.kernelci.org (node_memory_MemAvailable_bytes) <1000000000

Tips

  • Use diff(12h)>N to detect counter spikes — useful for error rates and request counters that only go up.
  • Use diff(12h)==0 to detect stalled counters (e.g. pipeline stopped processing jobs).
  • Label filters are exact match. For multiple labels, separate with commas.

PostgreSQL

Endpoint scheme: postgresql://user:password@host:port/database

Connects to the database and optionally runs a query. Connection failure → CRITICAL.

Examples

What to monitorEndpointSelectorCondition
Basic up/down postgresql://user:pass@db.kernelci.org/kcidb — —
Read-only check postgresql://user:pass@db.kernelci.org/kcidb SELECT pg_is_in_recovery()::int ==1
Table row count postgresql://user:pass@db.kernelci.org/kcidb SELECT count(*) FROM submissions WHERE created_at > now() - interval '1 hour' ==0
Replication lag postgresql://user:pass@db.kernelci.org/kcidb SELECT EXTRACT(EPOCH FROM replay_lag)::int FROM pg_stat_replication LIMIT 1 >300
Active connections postgresql://user:pass@db.kernelci.org/kcidb SELECT count(*) FROM pg_stat_activity WHERE state = 'active' >50

The Selector is the SQL query. The first column of the first row is the measured value. Leave empty for a simple SELECT 1 connectivity check.

Kubernetes

Endpoint scheme: k8s://namespace or k8s://namespace/label=value

Connects via the system kubeconfig. Lists pods and checks their uptime. WARNING if any pod has been running for less than 30 minutes (may indicate crash-looping or recent restarts).

Examples

What to monitorEndpointSelectorConditionNotes
All production pods k8s://production — — All pods in production namespace
API pods only k8s://production/app=api — — Only pods with label app=api
Pipeline workers k8s://production/component=pipeline-worker — — Filter by component label
Staging pods k8s://staging — — Staging namespace, non-critical

What it measures

  • Value: total number of matching pods.
  • State: WARNING if any pod uptime < 30 minutes, OK otherwise.
  • The message lists which pods have low uptime, e.g. api-7f8b9-xk2j4: uptime 3m.
  • Uses the system kubeconfig (typically ~/.kube/config or in-cluster credentials).

Monitoring pending pods via Prometheus

To monitor the number of pending pods or pod restart counts, use Prometheus metrics from kube-state-metrics instead:

WhatEndpointSelectorCondition
Pending pods promhttps://kube-state-metrics:8080 (kube_pod_status_phase,phase=Pending,namespace=production) >0
Pod restarts promhttps://kube-state-metrics:8080 (kube_pod_container_status_restarts_total,namespace=production,container=api) diff(12h)>5
Unavailable replicas promhttps://kube-state-metrics:8080 (kube_deployment_status_replicas_unavailable,deployment=api) >0

Docker via SSH

Endpoint scheme: ssh://user@host

SSHs into a remote host, runs docker ps, and parses container uptime. WARNING if any container has been running for less than 30 minutes. Uses the system SSH config (~/.ssh/config, keys, known_hosts).

Examples

What to monitorEndpointSelectorConditionNotes
All containers on host ssh://deploy@storage.kernelci.org — — Checks all running containers
Nginx only ssh://deploy@storage.kernelci.org nginx — Substring match on container name
Legacy storage ssh://deploy@legacy-storage.kernelci.org storage — Only containers with "storage" in name
KCIDB services ssh://deploy@kcidb.kernelci.org kcidb — All KCIDB-related containers

What it measures

  • Value: number of matching running containers.
  • State: WARNING if any container uptime < 30 minutes, CRITICAL if SSH connection fails or docker ps errors.
  • The message lists containers with low uptime.
  • SSH must be pre-configured — the daemon uses the system SSH agent and ~/.ssh/config.

Docker restart counts via Prometheus

To monitor container restart counts rather than just current uptime, expose metrics with cAdvisor or a Docker exporter and use a Prometheus check:

WhatEndpointSelectorCondition
Container restarts promhttps://storage.kernelci.org:9101 (container_restart_count,name=nginx) diff(12h)>3

Conditions & Alerts

Conditions are applied after the checker runs. They can only make the state worse (never improve it). If a checker already returns CRITICAL, a condition cannot downgrade it to WARNING.

Operators

ConditionTriggers WARNING whenExample use case
>NValue is greater than N>90 — disk usage above 90%
<NValue is less than N<7 — cert expires in less than 7 days
>=NValue is greater than or equal to N>=100 — connection pool full
<=NValue is less than or equal to N<=0 — no free space
==NValue equals N==1 — database in read-only mode
!=NValue does not equal N!=0 — error count is non-zero

Diff over time

Compare the current value against a historical measurement:

diff(12h)>100    — warn if value grew by more than 100 in the last 12 hours
diff(24h)<-50    — warn if value dropped by more than 50 in 24 hours
diff(12h)==0     — warn if value hasn't changed (stalled counter)

The daemon looks up the value recorded approximately that many hours ago (±30 minute tolerance window) and computes current - past.
